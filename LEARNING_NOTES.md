# Learning Notes

## Session 1 - Project Foundation

### Concepts Learned

- **Next.js App Router:** Files inside `src/app` define routes and layouts.
- **Server-first rendering:** App Router pages are server components unless
  they explicitly use `"use client"`.
- **Prisma schema:** `prisma/schema.prisma` describes database models and
  generates a typed TypeScript client.
- **Database connection URL:** The application can switch database environments
  by changing `DATABASE_URL`, without changing product code.
- **Reproducible builds:** Remote fonts made the initial build depend on Google
  Fonts, so the foundation uses system fonts instead.

### Decisions

- Use Next.js, TypeScript, Tailwind CSS, and Prisma.
- Use local Prisma Postgres during early development because Docker and a local
  PostgreSQL installation are not currently available.
- Keep the product schema out of Session 1. Design the `Tender` model carefully
  in Session 2.
- Keep `.env` private and commit only `.env.example`.

### Commands

```bash
npm run db:dev
npm run db:validate
npm run dev
npm run lint
npm run build
```

## Session 2 - First Tender Model

### Concepts Learned

- **Normalized columns:** Frequently displayed, searched, filtered, or matched
  fields have dedicated database columns.
- **Raw source payload:** The original Etimad list JSON is preserved in a
  `JSONB` column for debugging and future remapping.
- **Required versus optional:** Fields available on every list record are
  required. Detail-page fields are optional because availability varies.
- **Database types:** Dates use `DateTime`, money uses fixed-precision
  `Decimal`, long descriptions use `Text`, and raw JSON uses `Json`.
- **Unique constraints:** `referenceNumber` and `sourceTenderId` prevent the
  importer from creating duplicate tender records.
- **Indexes:** Indexes speed up expected filters and sorting, but each index
  also adds storage and write cost.
- **Migration:** A migration is version-controlled SQL that describes how to
  change a database schema.

### Model Groups

- Stable identity fields connect our record to Etimad.
- Searchable tender fields support display, filters, and relevance matching.
- Detail-enrichment fields store useful public detail-page information.
- Ingestion-tracking fields explain when and how the record was observed.

### Local Migration Note

The local Prisma Postgres server accepted `prisma db push`, but closed the
connection used by `prisma migrate dev` and `prisma migrate resolve`. The
exact SQL generated by `prisma migrate diff` was checked into
`prisma/migrations`, but the current local database cannot reliably maintain
Prisma migration metadata.

For current local development, synchronize the schema with:

```bash
npm run db:push
```

When Docker PostgreSQL is introduced, apply and verify checked-in migrations
with:

```bash
npx prisma migrate deploy
```

## Session 3 - Validate and Map One Etimad Page

### Concepts Learned

- **External data is untrusted:** A successful HTTP response can still contain
  HTML, malformed JSON, missing fields, or unexpected field types.
- **TypeScript is compile-time only:** A TypeScript type cannot validate JSON
  received while the application is running.
- **Runtime validation:** Zod checks the real response and returns trusted,
  typed data only after required fields pass validation.
- **Mapping:** External Etimad names and formats are converted into our
  application's naming and data conventions.
- **Timezone normalization:** Etimad timestamps without an offset are treated
  as Saudi time (`+03:00`) and converted into JavaScript `Date` values.
- **Content-type guard:** The fetcher rejects HTML traffic-protection responses
  before trying to parse them as JSON.

### Data Flow

```text
Etimad HTTP response
        ↓
unknown JSON
        ↓ Zod validation
trusted Etimad list record
        ↓ mapping
internal Tender preview shape
```

### Files

- `src/lib/etimad/list-schema.ts` validates Etimad's list-page response.
- `src/lib/etimad/fetch-list-page.ts` performs the read-only HTTP request.
- `src/lib/etimad/map-list-tender.ts` maps source fields into internal names.
- `scripts/preview-etimad-list.ts` previews three validated records.
- `src/lib/etimad/list-schema.test.ts` proves malformed data is rejected.

### Traffic-Protection Finding

A basic Node `fetch` received HTML instead of JSON. Etimad's public visitor
endpoint returned JSON when the request included the ordinary AJAX headers
used by its own visitor page. The fetcher checks the response content type and
will stop with a clear error if this behavior changes.

### Commands

```bash
npm run etimad:preview
npm test
```

The preview command is read-only and does not connect to PostgreSQL.

- The difference between validating and mapping data.
- Why an HTTP `200` response is not automatically trustworthy.
- Why timestamps need an explicit timezone assumption.
- Why the preview intentionally writes nothing to the database.

## Session 4 - Persist Tenders with Idempotent Upserts

### Concepts Learned

- **Persistence:** Validated and mapped records are now written to PostgreSQL
  through Prisma.
- **Idempotency:** Running the same import repeatedly reaches the same database
  state instead of creating duplicate tenders.
- **Upsert:** An upsert updates a matching record or inserts it when no match
  exists.
- **Stable identity:** Etimad's unique reference number is the importer's
  conflict key. The source tender ID remains separately unique.
- **Observation timestamps:** `firstSeenAt` records initial discovery, while
  `lastSeenAt` changes every time the importer observes the tender.
- **Field ownership:** List imports update list-owned fields but do not erase
  description or other fields that a later detail-page importer will own.
- **Database adapter:** Prisma 7 uses the official PostgreSQL driver adapter to
  connect the generated Prisma Client to PostgreSQL.

### Data Flow

```text
Etimad HTTP response
        ↓ validation and mapping
internal tender records
        ↓ upsert by referenceNumber
PostgreSQL Tender table
```

### Idempotency Proof

The importer was run twice against the same live Etimad page:

```text
First run:  24 created, 0 updated
Second run: 0 created, 24 updated
```

The second run did not increase the number of tender records.

### Files

- `src/lib/prisma.ts` creates the reusable Prisma client and PostgreSQL
  adapter.
- `src/lib/etimad/persist-list-tenders.ts` builds create/update data and
  performs transactional upserts.
- `scripts/import-etimad-list.ts` runs the first-page import and prints a
  summary.
- `src/lib/etimad/persist-list-tenders.test.ts` checks timestamp and field
  ownership behavior.

### Commands

```bash
npm run db:dev
npm run etimad:import
npm test
```

### You Should Be Able To Explain

- Why the list importer must preserve detail-page fields.
- Why the importer uses a unique reference number as its upsert key.
- What the PostgreSQL driver adapter does for Prisma Client.

## Session 5 - Complete Phase 1 with the Tender List

### Concepts Learned

- **Sequential pagination:** The importer fetches five Etimad pages one after
  another with a short delay, instead of sending a burst of simultaneous
  requests.
- **Cross-page deduplication:** Records are deduplicated by reference number
  before they reach the persistence layer.
- **Server component data fetching:** The `/tenders` page queries PostgreSQL
  directly through Prisma while rendering on the server.
- **Dynamic rendering:** The tender page is rendered on demand so it reflects
  the current local database rather than becoming a build-time snapshot.
- **Mixed-language interface:** Product navigation and labels are English,
  while original Arabic tender content uses explicit right-to-left direction.
- **Source transparency:** Every card links back to its original Etimad page.

## Session 6 - Tender Detail Enrichment and Detail Page

### Concepts Learned

- **HTML parsing:** Etimad detail endpoints return HTML rather than JSON, so
  Cheerio converts the markup into a traversable document structure.
- **Label/value extraction:** The parser reads recurring Etimad label and value
  classes instead of depending on fragile element positions.
- **Raw snapshots:** The basic page and each public detail component are stored
  unchanged for debugging when Etimad markup changes.
- **Normalized enrichment:** Useful detail fields are written into dedicated
  `Tender` columns for display and future search or matching.
- **Enrichment status:** Each tender records whether public detail enrichment
  is pending, complete, or failed, including the latest error.
- **Dynamic route:** `/tenders/[id]` fetches one stored tender using the
  identifier in the URL.
- **Attachment metadata:** Public attachment names and source links are stored
  without automatically downloading untrusted files.

### Detail Data Flow

```text
Stored tender
    ↓ manual enrichment
Etimad basic detail page + public HTML components
    ↓ structured HTML parsing
normalized detail fields + raw snapshots + attachment metadata
    ↓
PostgreSQL → /tenders/[id]
```

### Real-Data Proof

```text

Execution city: جدة
Raw snapshot size: approximately 320 KB
```

The same tender was enriched again successfully, proving that detail

- `src/lib/etimad/parse-tender-detail.ts` parses and normalizes public fields.

- `src/app/tenders/[id]/page.tsx` renders the internal tender detail page.

## Session 7 - Search, Filters, Sorting, and Pagination

### Concepts Learned

- **Query parameters:** Search state is represented in the URL, such as
  `/tenders?q=تقنية&deadline=7`, making filtered views refreshable and
  shareable.
- **Server-side filtering:** Prisma translates search choices into PostgreSQL
  conditions instead of loading every tender and filtering in the browser.
- **Partial versus exact matching:** Keywords use partial matching across
  titles, descriptions, agencies, activities, and identifiers. Dropdown
  filters use exact stored values.
- **Combined conditions:** Multiple active filters are joined with `AND`, while
  keyword-search fields are joined with `OR`.
- **Sorting:** Results can be ordered by publication date or submission
  deadline, with missing deadlines placed last.
- **Pagination:** The browser fetches 24 matching tenders per page and preserves
  active filters in previous/next links.
- **Closing-soon window:** Deadline filters compare submission dates with the
  current date and a seven- or thirty-day future boundary.

### Search Data Flow

```text
/tenders query parameters
        ↓ validation and normalization
Prisma where + orderBy + skip + take
        ↓
PostgreSQL filtered results and count
        ↓
shareable paginated tender browser
```

### Real-Data Proof

Live verification against the local 120-tender dataset confirmed:

```text
Default page: 24 tender cards
Page 2: 24 tender cards with previous/next navigation
Reference search 260639003513: 1 result
Execution region مكة المكرمة: 1 result
```

### Files

- `src/lib/tenders/search.ts` validates query parameters and builds Prisma
  filtering and sorting objects.
- `src/lib/tenders/search.test.ts` verifies parameter normalization,
  closing-soon conditions, and deadline sorting.
- `src/app/tenders/page.tsx` renders controls, filtered results, and pagination.

### You Should Be Able To Explain

- Why search state belongs in URL query parameters.
- Why database filtering scales better than browser-only filtering.
- Why keyword fields use `OR` inside the wider `AND` filter.
- How `skip` and `take` implement pagination.
- Why missing submission deadlines are sorted last.
- Why region filtering currently applies only to enriched tenders.

## Session 8 - Save, Ignore, and Tender Notes

### Concepts Learned

- **Relational ownership:** Etimad source fields remain on `Tender`, while
  local user decisions live in a separate one-to-one `TenderDecision` record.
- **Nullable decision status:** A decision can be saved, ignored, or neutral.
  Notes can exist independently of the current status.
- **Server actions:** Forms call server-side functions that validate form data,
  write through Prisma, and revalidate affected pages.
- **Reversible workflow:** Save and Ignore can replace each other or be cleared
  with Undo.
- **Default exclusion:** Ignored tenders are excluded from the main browser at
  the database-query level.
- **Decision workspace:** Saved and ignored tenders remain reviewable on a
  dedicated page, including stored notes.

### Decision Data Flow

```text
Save / Ignore / Undo / Note form
        ↓ server action validation
TenderDecision upsert
        ↓ page revalidation
tender browser + detail page + decision workspace
```

### Real-Data Proof

Temporary saved and ignored decisions were created against real local tenders.
Verification confirmed:

```text
Ignored tender hidden from default browser
Saved tender visible in saved workspace
Ignored tender visible and reversible in ignored workspace
Stored note visible in saved workspace and detail page
```

The temporary verification decisions were removed afterward.

### Files

- `src/app/tenders/actions.ts` validates and persists decision and note forms.
- `src/app/tenders/decision-controls.tsx` provides reusable Save, Ignore, and
  Undo controls.
- `src/app/tenders/saved/page.tsx` displays saved and ignored tenders.
- `src/app/tenders/page.tsx` hides ignored results and shows decision controls.
- `src/app/tenders/[id]/page.tsx` supports decisions and tender notes.

### You Should Be Able To Explain

- Why local decisions should not be stored as Etimad source fields.
- Why one tender has at most one local decision record in this single-user
  prototype.
- How a server action receives and validates form data.
- Why ignored tenders are hidden in the database query rather than CSS.
- Why Undo is necessary for a reversible decision workflow.

## Session 9 - Company Profile

### Concepts Learned

- **Product profile modeling:** A lightweight company profile captures what the
  company does and what opportunities it wants without requiring sensitive
  documents.
- **PostgreSQL arrays:** Services, activities, industries, entities, regions,
  keywords, exclusions, and opportunity types are stored as typed string
  arrays.
- **Form normalization:** Comma- and newline-separated text is trimmed,
  deduplicated, and converted into arrays before validation.
- **Zod form validation:** Required identity fields and all preference arrays
  are validated before database writes.
- **Singleton prototype record:** The current local single-user app upserts one
  `primary` company profile. Multi-company ownership can be introduced with
  authentication later.
- **Editable preferences:** Stored profile data is loaded back into the form so
  users can revise matching inputs over time.

### Profile Data Flow

```text
Company profile form
        ↓ normalize comma/newline lists
Zod validation
        ↓
CompanyProfile upsert
        ↓
editable structured matching inputs
```

### Real-Data Proof
